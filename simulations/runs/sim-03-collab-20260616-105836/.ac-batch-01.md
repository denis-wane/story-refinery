# Acceptance Criteria: INFRA-001 — WebSocket Service Capacity Spike

## Refined Story Statement
As an **engineering team**, I want to load-test our existing WebSocket infrastructure at realistic document-editing traffic patterns, so that we make an evidence-based decision on whether to extend the existing service or provision a dedicated one before committing to the sync architecture.

## Assumptions
- The existing WebSocket service is accessible in a staging or load-test environment — **Confirmed**
- "Editing frequency" can be modeled as approximately 5 keystrokes/second per active editor plus cursor-move events every 100ms — **Unconfirmed**
- The 500ms SLA is scoped to same-region traffic only — **Unconfirmed**
- The load test environment can reach the same WebSocket tier as production — **Confirmed**

Unconfirmed assumptions are listed in Open Questions.

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-1: CRDT vs. OT framework | Load test is framework-agnostic (measures raw WS throughput/latency); CRDT/OT decision deferred post-spike | Open Question |
| G-2: WebSocket infrastructure adequacy | This story directly resolves G-2 by producing a written recommendation | AC-1 through AC-5 |
| G-4: Maximum document size | Document size affects message payload size; spike must test with representative payload sizes (1KB–50KB messages) | AC-3 |

## Acceptance Criteria

### AC-1: Load Test Executes Successfully Against Staging WS Service
**Given** the staging WebSocket service is running and accessible
**When** the load test harness connects simulated editing clients at the target concurrency level
**Then** the test runs to completion without the harness itself erroring out, and raw connection/message logs are written to a retrievable artifact

**Category:** happy-path
**Priority:** must-have

### AC-2: p50 and p95 Propagation Latency Measured and Reported
**Given** the load test is running with ≥ 50 concurrent simulated editor connections
**When** each simulated client sends an edit event (keystroke payload) and a timestamp
**Then** the harness records round-trip latency per message and produces a report showing p50 and p95 latency in milliseconds, sampled over a minimum 5-minute test window

**Category:** happy-path
**Priority:** must-have

### AC-3: Representative Payload Sizes Are Tested
**Given** document payloads vary in size
**When** the load test runs
**Then** it exercises three payload tiers: small (1KB), medium (10KB), and large (50KB) messages, and the report shows latency broken down by tier

**Category:** boundary
**Priority:** must-have

### AC-4: Maximum Concurrent Connection Limit Identified
**Given** the load test ramps concurrency incrementally (10, 25, 50, 100, 200 simultaneous connections)
**When** latency at any ramp level exceeds 500ms p95 OR the service begins dropping connections
**Then** the test records the concurrency level at which degradation begins and includes this threshold in the written recommendation

**Category:** boundary
**Priority:** must-have

### AC-5: Written Recommendation Produced
**Given** the load test has completed and metrics are collected
**When** the engineering team reviews the results
**Then** a written recommendation document exists that states one of: (a) "Existing service sufficient — extend with Yjs bindings", or (b) "Dedicated service required — existing service cannot meet 500ms SLA at target concurrency"; the document includes the p95 latency figures, the connection limit found, and a rationale

**Category:** happy-path
**Priority:** must-have

### AC-6: Test Scope Does Not Touch Production
**Given** the load test targets staging infrastructure
**When** the test runs
**Then** no production WebSocket connections are affected, and the test report confirms the target environment was staging

**Category:** error-handling
**Priority:** must-have

### AC-AUTH-1: Unauthenticated Access to Load Test Results Rejected
**Given** no valid authentication token is present
**When** a request is made to retrieve the load test report artifact or test logs
**Then** the system returns 401 Unauthorized

**Category:** security
**Priority:** must-have

### AC-AUTH-2: Insufficient Permissions to Trigger Load Test Rejected
**Given** an authenticated user without the `infra:load-test:run` permission
**When** a request is made to trigger or schedule a load test run
**Then** the system returns 403 Forbidden with a message identifying the missing permission

**Category:** security
**Priority:** must-have

## Non-Functional Requirements

### Error Handling
| Scenario | Expected Behavior | Priority |
|----------|------------------|----------|
| Staging WS service unreachable | Load test harness exits with a clear error message: "Could not connect to staging WS at [host:port] — aborting"; no partial report written | must-have |
| Test harness loses connection mid-run | Harness logs the disconnection event with timestamp, attempts one reconnect, records the gap in the timeline, and continues; final report notes the interruption | must-have |
| Simulated client error rate > 5% | Report flags this and recommendation notes unreliable baseline | should-have |

### Performance
- **Test duration:** Minimum 5-minute sustained load window after ramp-up
- **Ramp-up:** 60-second ramp from 0 to target concurrency
- **Reporting latency:** Report generated within 5 minutes of test completion

### Security
- **Access control:** Load test tooling must require authentication; results stored in access-controlled artifact store
- **No production targeting:** Configuration must explicitly require a `--env staging` flag; absence of flag causes harness to refuse to run

### Accessibility
- Not applicable (engineering tooling, no UI)

## Open Questions
1. What is the realistic keystroke/cursor-event frequency per editor? This must be agreed before the test model is considered representative.
2. Is the 500ms SLA same-region only, or does it apply globally? If global, the load test scope must include cross-region latency measurement.
3. Is there an existing load-testing framework (k6, Gatling, etc.) already in use that this spike should use for consistency?

# Acceptance Criteria: INFRA-002 — Dedicated y-websocket Service

## Refined Story Statement
As an **editor**, I want document changes routed through a dedicated WebSocket service backed by Yjs awareness, so that the collaborative editing path is isolated from general notification traffic and can be scaled independently.

## Assumptions
- Yjs (`y-websocket`) is the chosen sync library — **Unconfirmed** (pending G-1 resolution; this story assumes Yjs per story scope)
- A single-region deployment is sufficient for v1 — **Confirmed**
- The existing document store (database) is accessible from the new service's deployment environment — **Confirmed**
- Health check endpoint follows the existing service's `/health` convention — **Unconfirmed**

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-1: CRDT vs. OT framework | Story explicitly scopes to Yjs/y-websocket; if architectural decision selects OT instead, this story must be re-scoped | Open Question (OQ-1) |
| G-2: WebSocket infrastructure | This story implements the decision from INFRA-001; scope is blocked until INFRA-001 recommendation is accepted | AC-1 |
| G-4: Maximum document size | Yjs document persistence must handle variable document sizes; max payload not yet set; service must not impose an artificial cap below any future document size limit | AC-4 (size guard) |

## Acceptance Criteria

### AC-1: y-websocket Service Starts and Accepts Connections
**Given** the y-websocket service is deployed and running
**When** a Tiptap/Yjs client connects using a valid document room ID and auth token
**Then** the connection is established within 2 seconds, and the server sends the current document state to the connecting client

**Category:** happy-path
**Priority:** must-have

### AC-2: Edits Persist to Document Store on Change
**Given** an editor is connected to the y-websocket service and makes changes
**When** the Yjs document state updates
**Then** the updated document state is persisted to the existing document store within 5 seconds of the change, confirmed by reading back from the store

**Category:** happy-path
**Priority:** must-have

### AC-3: Service Isolated From General Notification WebSocket Traffic
**Given** the dedicated y-websocket service is running alongside the general WS service
**When** traffic spikes occur on the general notification WS service
**Then** p95 latency on the y-websocket service remains unaffected (within 10% of baseline measured in INFRA-001)

**Category:** edge-case
**Priority:** must-have

### AC-4: Large Document State Handled Without Rejection
**Given** a document with a Yjs binary update payload up to 50KB
**When** a client connects and the server sends the initial document state
**Then** the full state is delivered without truncation or error

**Category:** boundary
**Priority:** must-have

### AC-5: Health Check Endpoint Returns Correct Status
**Given** the y-websocket service is running normally
**When** a GET request is made to `/health`
**Then** the endpoint returns HTTP 200 with a JSON body containing `{"status": "ok", "connections": <current_count>}`

**Given** the y-websocket service cannot reach the document store
**When** a GET request is made to `/health`
**Then** the endpoint returns HTTP 503 with `{"status": "degraded", "reason": "document store unreachable"}`

**Category:** happy-path / error-handling
**Priority:** must-have

### AC-6: Service Restarts Automatically After Crash
**Given** the y-websocket service process exits unexpectedly
**When** the process manager (e.g., systemd, Kubernetes deployment) detects the exit
**Then** the service restarts within 30 seconds, and reconnecting clients receive the current document state (no data loss from the document store)

**Category:** error-handling
**Priority:** must-have

### AC-7: Unauthorized Client Connection Rejected at Service Level
**Given** a client presents an invalid or missing auth token
**When** it attempts to open a WebSocket connection to the y-websocket service
**Then** the service closes the WebSocket with code 4001 (Unauthorized) and does not relay any document state

**Category:** security
**Priority:** must-have

### AC-8: Unknown Document Room Handled Gracefully
**Given** a client requests a room ID that does not correspond to any existing document
**When** the connection is established
**Then** the service initializes an empty Yjs document for that room ID and returns it; it does NOT return 404 or close the connection (room creation on first connect is expected behavior)

**Category:** edge-case
**Priority:** must-have

### AC-AUTH-1: Unauthenticated Access Rejected
**Given** no valid authentication token is present
**When** a request is made to the y-websocket service management endpoints (e.g., `/health`, admin endpoints)
**Then** the system returns 401 Unauthorized

**Category:** security
**Priority:** must-have

### AC-AUTH-2: Insufficient Permissions Rejected
**Given** an authenticated user without document access permission
**When** a WebSocket connection is attempted to a document room
**Then** the service closes the connection with code 4003 (Forbidden) and logs the attempt with the user ID and document ID

**Category:** security
**Priority:** must-have

## Non-Functional Requirements

### Error Handling
| Scenario | Expected Behavior | Priority |
|----------|------------------|----------|
| Document store write fails | Service retries the persist operation 3× with exponential backoff (1s, 2s, 4s); if all retries fail, logs an error with document ID and continues serving live connections without dropping them | must-have |
| Client sends malformed Yjs binary update | Service discards the update, logs a warning with client ID and document ID, and does not propagate to other clients | must-have |
| All clients disconnect from a room | Room is kept in memory for 60 seconds, then garbage-collected; document state remains in the store | should-have |

### Performance
- **Connection establishment:** < 2s p95 under 100 concurrent new connections
- **Message relay latency:** p95 ≤ 500ms same-region (acceptance tested as part of SYNC-002)
- **Persist lag:** Yjs updates written to document store within 5 seconds of receipt

### Security
- **Authentication:** Every WebSocket connection must present a valid session token validated against the auth service
- **Room isolation:** A client authenticated for document A must not be able to connect to document B's room without permission
- **Input validation:** Yjs binary updates are validated for format before relay; malformed payloads are dropped

### Accessibility
- Not applicable (backend service, no UI)

## Open Questions
1. (OQ-1) If the architectural decision from INFRA-001 selects an OT-based system instead of Yjs, this story must be re-scoped entirely. Confirm Yjs selection before implementation begins.
2. What is the restart budget? How many consecutive crashes within what window triggers an alert vs. a page?
3. Is the "existing document store" a relational DB, an object store, or both? The persistence mechanism in AC-2 depends on this.

# Acceptance Criteria: INFRA-003 — Document-Level Permission Table

## Refined Story Statement
As an **application**, I want a dedicated `document_permissions` table storing per-user view/comment/edit grants for each document, so that document-level permissions are additive over workspace roles without coupling to or modifying the existing RBAC schema.

## Assumptions
- The existing RBAC system is not document-aware (grants are workspace/org level only) — **Confirmed**
- Permission levels are exactly three: `view`, `comment`, `edit` — **Confirmed**
- A single user can have at most one permission level per document (the highest wins; no stacking) — **Unconfirmed**
- The `document_permissions` table lives in the same database as the existing RBAC tables — **Unconfirmed**
- Document ownership (creator) is stored elsewhere (e.g., a `documents` table); this table stores explicit grants only — **Confirmed**

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-5: Existing permissions model integration | This story creates the additive overlay table without modifying the existing RBAC schema; integration logic (how existing roles interact with document grants) is deferred to PERM-003 | AC-1 (schema), Open Question (OQ-1) |

## Acceptance Criteria

### AC-1: `document_permissions` Table Created by Migration
**Given** the migration is run in a clean environment
**When** the migration completes successfully
**Then** a `document_permissions` table exists with at minimum the columns: `id` (PK), `document_id` (FK to documents), `user_id` (FK to users), `permission_level` (enum: `view`, `comment`, `edit`), `granted_at` (timestamp), `granted_by` (user_id FK); a unique constraint exists on `(document_id, user_id)`

**Category:** happy-path
**Priority:** must-have

### AC-2: Migration Is Reversible
**Given** the migration has been applied
**When** the migration is rolled back
**Then** the `document_permissions` table is dropped and the schema returns to its pre-migration state with no orphaned constraints

**Category:** edge-case
**Priority:** must-have

### AC-3: Upsert Permission Grant for a User
**Given** a document ID and a user ID
**When** a PUT request is made to `PUT /documents/{doc_id}/permissions/{user_id}` with body `{"permission_level": "edit"}`
**Then** the service upserts the record (insert if new, update if existing) and returns HTTP 200 with the current permission record; the `granted_by` field is set to the requesting user's ID

**Category:** happy-path
**Priority:** must-have

### AC-4: Retrieve All Permissions for a Document
**Given** a document ID with multiple permission grants
**When** a GET request is made to `GET /documents/{doc_id}/permissions`
**Then** the response returns HTTP 200 with an array of all permission records for that document, each including `user_id`, `permission_level`, `granted_at`, `granted_by`

**Category:** happy-path
**Priority:** must-have

### AC-5: Retrieve Permission for a Specific User on a Document
**Given** a document ID and a user ID that has an existing permission record
**When** a GET request is made to `GET /documents/{doc_id}/permissions/{user_id}`
**Then** the response returns HTTP 200 with the single permission record

**Given** the user has no record for that document
**When** the same GET request is made
**Then** the response returns HTTP 404 (no implicit default grant)

**Category:** happy-path / edge-case
**Priority:** must-have

### AC-6: Delete Permission Grant
**Given** a user has an existing permission record for a document
**When** a DELETE request is made to `DELETE /documents/{doc_id}/permissions/{user_id}`
**Then** the record is removed, the response is HTTP 204, and subsequent GET for that user returns 404

**Category:** happy-path
**Priority:** must-have

### AC-7: Invalid Permission Level Rejected
**Given** a PUT request to grant permissions
**When** the request body contains `{"permission_level": "superadmin"}` or any value not in `[view, comment, edit]`
**Then** the service returns HTTP 400 with a message: "Invalid permission_level. Must be one of: view, comment, edit"

**Category:** error-handling
**Priority:** must-have

### AC-8: Duplicate Grant Upserts Without Error
**Given** a user already has `edit` permission on a document
**When** a PUT request grants the same user `comment` permission on the same document
**Then** the service updates the record to `comment` and returns HTTP 200 (no unique constraint violation error is surfaced to the caller)

**Category:** edge-case
**Priority:** must-have

### AC-9: Non-Existent Document Rejected
**Given** a document ID that does not exist in the `documents` table
**When** a PUT request attempts to grant a permission for that document ID
**Then** the service returns HTTP 404 with message "Document not found"

**Category:** error-handling
**Priority:** must-have

### AC-AUTH-1: Unauthenticated Access Rejected
**Given** no valid authentication token is present
**When** a request is made to any `/documents/{doc_id}/permissions` endpoint
**Then** the system returns 401 Unauthorized

**Category:** security
**Priority:** must-have

### AC-AUTH-2: Insufficient Permissions Rejected
**Given** an authenticated user who is not the document owner and does not have the `document:manage-permissions` privilege
**When** a PUT or DELETE request is made to modify permissions on a document
**Then** the system returns 403 Forbidden with a message identifying the missing privilege

**Category:** security
**Priority:** must-have

## Non-Functional Requirements

### Error Handling
| Scenario | Expected Behavior | Priority |
|----------|------------------|----------|
| Non-existent user_id in grant request | Return HTTP 404 "User not found" | must-have |
| Database unavailable during permission write | Return HTTP 503 "Service temporarily unavailable"; do not partially write | must-have |
| Concurrent upserts for same (doc_id, user_id) | Database unique constraint handles race; last write wins; no 500 error surfaced | must-have |

### Performance
- **Read latency:** `GET /documents/{doc_id}/permissions` returns within 100ms p95 for documents with up to 1,000 permission records
- **Write latency:** Upsert completes within 200ms p95

### Security
- **Authorization:** Only document owners and users with `document:manage-permissions` privilege can write permissions; any authenticated user with at least `view` on the document can read permissions for that document
- **Input validation:** `document_id` and `user_id` must be valid UUIDs; reject otherwise with HTTP 400
- **Audit:** Every write to `document_permissions` is logged with `granted_by`, timestamp, and old/new values

### Accessibility
- Not applicable (backend API, no UI)

## Open Questions
1. (OQ-1) How does an implicit workspace-level role (e.g., "workspace admin") interact with the absence of a `document_permissions` record? Is a missing record treated as "no access" or "inherit from workspace role"? This must be decided before PERM-003 can enforce access correctly.
2. Is `document_id` a UUID foreign key, or could it be an integer? The schema must match the existing `documents` table type.
3. Should `granted_by` be nullable for system-generated grants (e.g., when a document is created and the creator is auto-granted `edit`)?

# Acceptance Criteria: RICH-001 — Tiptap Editor With Inline Formatting

## Refined Story Statement
As an **editor**, I want to apply inline formatting — bold, italic, underline, headings (H1–H3), and hyperlinks — to selected text in the document, so that I can create structured, readable content without switching to a separate tool.

## Assumptions
- Tiptap (ProseMirror-based) is the chosen editor framework — **Confirmed**
- Toolbar buttons are visible and accessible; keyboard shortcuts are an additive affordance — **Confirmed**
- The link insert dialog accepts only valid HTTP/HTTPS URLs — **Unconfirmed**
- Headings H1–H3 are the only supported heading levels in v1 — **Confirmed**
- Formatting state is stored in the Yjs/Tiptap document model even before sync is wired (SYNC-001) — **Confirmed**

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-1: CRDT vs. OT framework | Tiptap supports both Yjs and ShareDB; inline formatting marks are stored in the editor document model regardless; this AC does not depend on sync algorithm choice | AC-1 (editor init) |
| G-4: Maximum document size | No document size limit is enforced at the formatting layer; this AC defers size enforcement to RICH-004 and infrastructure stories | Open Question (OQ-2) |

## Acceptance Criteria

### AC-1: Editor Mounts and Accepts Text Input
**Given** a user navigates to a document in the editor
**When** the Tiptap editor component mounts
**Then** an empty editable canvas is rendered, the user can click into it and type text, and the text appears in the editor

**Category:** happy-path
**Priority:** must-have

### AC-2: Bold Formatting Applied and Removed via Toolbar
**Given** text is selected in the editor
**When** the user clicks the Bold toolbar button (or presses Ctrl+B / Cmd+B)
**Then** the selected text renders in bold; clicking Bold again (or pressing the shortcut again) removes bold from the selection

**Category:** happy-path
**Priority:** must-have

### AC-3: Italic Formatting Applied and Removed
**Given** text is selected in the editor
**When** the user clicks the Italic toolbar button or presses Ctrl+I / Cmd+I
**Then** the selected text renders in italic; repeating the action removes italic

**Category:** happy-path
**Priority:** must-have

### AC-4: Underline Formatting Applied and Removed
**Given** text is selected in the editor
**When** the user clicks the Underline toolbar button or presses Ctrl+U / Cmd+U
**Then** the selected text renders underlined; repeating the action removes underline

**Category:** happy-path
**Priority:** must-have

### AC-5: Headings H1–H3 Applied via Toolbar
**Given** the cursor is in a paragraph or an existing heading
**When** the user selects H1, H2, or H3 from the heading dropdown in the toolbar
**Then** the current block converts to the selected heading level and renders with the appropriate visual hierarchy (H1 largest, H3 smallest)

**Given** the user selects "Paragraph" from the same dropdown
**When** the current block is a heading
**Then** the block converts back to a plain paragraph

**Category:** happy-path
**Priority:** must-have

### AC-6: Hyperlink Inserted via Dialog
**Given** text is selected in the editor
**When** the user clicks the Link toolbar button
**Then** a dialog opens with a URL input field and Confirm/Cancel buttons

**Given** the user enters a valid URL (starting with `https://` or `http://`) and clicks Confirm
**When** the dialog is submitted
**Then** the selected text becomes a clickable hyperlink; hovering the link shows a tooltip with the URL; clicking the link in a non-edit context (or with Ctrl/Cmd+click in editor) opens the URL in a new tab

**Category:** happy-path
**Priority:** must-have

### AC-7: Invalid URL Rejected in Link Dialog
**Given** the link dialog is open
**When** the user enters a value that is not a valid HTTP/HTTPS URL (e.g., `javascript:alert(1)`, plain text, empty string) and clicks Confirm
**Then** the dialog displays an inline error "Please enter a valid URL starting with https:// or http://" and does not create a link

**Category:** error-handling / security
**Priority:** must-have

### AC-8: Existing Link Can Be Edited
**Given** text with an existing hyperlink is selected
**When** the user clicks the Link toolbar button
**Then** the dialog opens pre-populated with the existing URL, allowing the user to modify and confirm or remove the link

**Category:** edge-case
**Priority:** must-have

### AC-9: Formatting Combinations Render Correctly
**Given** text is selected
**When** the user applies both Bold and Italic
**Then** the text renders as bold-italic simultaneously; each mark is independently togglable without removing the other

**Category:** edge-case
**Priority:** must-have

### AC-10: Formatting Does Not Apply to Zero-Width Selections
**Given** no text is selected (cursor is positioned but nothing is highlighted)
**When** the user clicks a formatting toolbar button
**Then** the toolbar button state toggles (indicating the mark will apply to the next characters typed), but no existing text is re-formatted

**Category:** edge-case
**Priority:** should-have

### AC-AUTH-1: Unauthenticated Access Rejected
**Given** no valid authentication token is present
**When** a request is made to load a document in the editor
**Then** the system returns 401 Unauthorized and the editor canvas does not render

**Category:** security
**Priority:** must-have

### AC-AUTH-2: Insufficient Permissions Rejected
**Given** an authenticated user with only `view` or `comment` permission on the document
**When** the user attempts to type in or apply formatting to the editor canvas
**Then** the editor renders in read-only mode; toolbar formatting buttons are disabled; the system does not silently allow writes

**Category:** security
**Priority:** must-have

## Non-Functional Requirements

### Error Handling
| Scenario | Expected Behavior | Priority |
|----------|------------------|----------|
| Editor fails to mount (JS error) | Error boundary renders "Editor failed to load — please refresh" with a reload button; error is logged to the error tracking service | must-have |
| Toolbar button clicked with no editor focus | No action taken; no JS error thrown | must-have |

### Performance
- **Editor mount time:** Editor renders and is interactive within 1,500ms of page load (p95, on a 10KB document)
- **Formatting keystroke latency:** Formatting mark applied within one animation frame (16ms) of the keyboard shortcut being pressed

### Security
- **Link URL validation:** All link URLs must pass an allow-list check (`https://`, `http://` only); `javascript:`, `data:`, and protocol-relative URLs are rejected both in the dialog and on paste
- **Content sanitization:** Rich text output must be sanitized before persistence to prevent stored XSS; no raw HTML is accepted from client

### Accessibility
- All toolbar buttons must have `aria-label` attributes with descriptive names (e.g., "Bold", "Italic", "Insert link")
- Toolbar buttons must be keyboard-navigable (Tab/Shift+Tab) and activatable via Enter/Space
- Active formatting state must be communicated via `aria-pressed` on toggle buttons

## Open Questions
1. Should links open in a new tab unconditionally, or should external links open in a new tab and internal links navigate in the same tab?
2. (OQ-2) Is there a maximum character count per document? Without this bound, the editor has no guard against extremely large documents degrading performance.

# Acceptance Criteria: RICH-002 — Block-Level Elements: Lists and Code Blocks

## Refined Story Statement
As an **editor**, I want to insert bullet lists, numbered lists, and fenced code blocks into a document, so that I can represent structured content and technical material with appropriate formatting.

## Assumptions
- Tiptap extensions `BulletList`, `OrderedList`, and `CodeBlock` are used — **Confirmed**
- Markdown-shortcut triggers are enabled in the Tiptap configuration — **Confirmed**
- Code blocks do not require syntax highlighting in v1 — **Confirmed**
- The language selector for code blocks accepts freeform text input, not a constrained dropdown — **Unconfirmed**
- List items support basic inline formatting (bold/italic) — **Confirmed** (inherits from RICH-001)

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-1: CRDT vs. OT framework | Block-level elements are stored in the Tiptap document model; CRDT/OT choice does not affect this story's AC | Out of Scope for this story |
| G-4: Maximum document size | No size enforcement at block element level; deferred to RICH-004/infra stories | Out of Scope for this story |

## Acceptance Criteria

### AC-1: Bullet List Inserted via Toolbar
**Given** the cursor is in any block
**When** the user clicks the Bullet List toolbar button
**Then** the current block converts to a bullet list item, and subsequent Enter key presses create additional list items at the same level

**Category:** happy-path
**Priority:** must-have

### AC-2: Bullet List Created via Markdown Shortcut
**Given** the cursor is at the start of an empty line
**When** the user types `- ` (hyphen, space)
**Then** the block immediately converts to a bullet list item without inserting the literal `- ` characters

**Category:** happy-path
**Priority:** must-have

### AC-3: Numbered List Inserted via Toolbar
**Given** the cursor is in any block
**When** the user clicks the Ordered List toolbar button
**Then** the current block converts to a numbered list item starting at `1.`, and subsequent Enter key presses create sequentially numbered items

**Category:** happy-path
**Priority:** must-have

### AC-4: Numbered List Created via Markdown Shortcut
**Given** the cursor is at the start of an empty line
**When** the user types `1. ` (one, period, space)
**Then** the block converts to a numbered list item without inserting the literal characters

**Category:** happy-path
**Priority:** must-have

### AC-5: List Item Indented and Outdented
**Given** a list item exists

**Given** the item is already at the top level

**Category:** edge-case
**Priority:** must-have

### AC-6: List Exited with Double Enter on Empty Item

**Then** the empty list item is removed and a plain paragraph is inserted below the list

**Category:** edge-case
**Priority:** must-have

### AC-7: Code Block Inserted via Toolbar
**Given** the cursor is in any block
**When** the user clicks the Code Block toolbar button
**Then** a monospace-rendered code block is inserted, and text typed inside it is not subjected to inline formatting (bold/italic marks are inactive inside a code block)

**Category:** happy-path
**Priority:** must-have

### AC-8: Code Block Created via Markdown Shortcut

# Acceptance Criteria: CONF-001 — Automatic Merge and Same-Range Last-Writer-Wins Behavior

## Refined Story Statement
As an **editor**, I want non-overlapping concurrent edits to merge automatically and same-range concurrent edits to resolve by applying the last-received edit, so that I never encounter a manual merge conflict during real-time collaboration.

## Assumptions
- Yjs is the CRDT implementation (per SYNC-001) — **Confirmed**
- "Last writer" for same-range edits is determined by server relay order, not client timestamp — **Confirmed**
- Undo history is preserved for both editors involved in a same-range conflict — **Confirmed**
- No conflict resolution UI is exposed to users in v1 — **Confirmed**
- Offline reconnection uses the same last-writer-wins strategy as real-time (G-6 resolved by assumption) — **Confirmed** (see Assumption note below)
- Offline changes are treated as lower priority than concurrent server state on reconnect — **Confirmed** (per G-6 assumption; see OQ-1)

## Gap Traceability

| Gap | Resolution | Reference |
|-----|-----------|-----------|
| G-1: CRDT vs. OT framework | This story depends entirely on Yjs being selected (SYNC-001); Yjs's CRDT semantics naturally merge non-overlapping edits; same-range behavior is configured via server relay ordering | AC-1 (non-overlapping), AC-2 (same-range) |
| G-6: Offline + same-range conflict strategy | Resolved by assumption: last-writer-wins applied consistently on reconnect; offline edits treated as lower priority | AC-3 (reconnect behavior), Open Question (OQ-1) |

## Acceptance Criteria

### AC-1: Non-Overlapping Concurrent Edits Merge Automatically
**Given** Editor A is editing Paragraph 1 and Editor B is simultaneously editing Paragraph 2 of the same document
**When** both editors submit their changes within the same 500ms window
**Then** both editors' documents converge to a state containing both changes (Editor A sees Editor B's paragraph 2 edit; Editor B sees Editor A's paragraph 1 edit); no characters from either edit are lost or duplicated

**Category:** happy-path
**Priority:** must-have

### AC-2: Same-Range Concurrent Edit Resolves to Last-Received Edit
**Given** Editor A and Editor B both edit the same character range simultaneously (e.g., both replace the word "hello" at position 5–9)
**When** the y-websocket server relays both updates (Editor A's update arrives at the server first, Editor B's second)
**Then** both editors' documents converge to Editor B's version of that range (last-received wins per server relay order); no characters are duplicated or corrupted at the conflict site

**Category:** happy-path
**Priority:** must-have

### AC-3: Undo History Preserved for Both Editors After Same-Range Conflict
**Given** a same-range conflict has been resolved (Editor B's edit won)
**When** Editor A presses Ctrl+Z / Cmd+Z (undo)
**Then** Editor A's own edit is undone in their view (reverting to the pre-edit state for their undo stack); Editor B's winning edit remains visible; Editor A's undo does not remove Editor B's change

**Category:** edge-case
**Priority:** must-have

### AC-4: Offline Edit Merged on Reconnect (Non-Overlapping Case)
**Given** Editor A goes offline and edits Paragraph 3 while Editor B (online) edits Paragraph 4
**When** Editor A reconnects
**Then** both documents converge to contain both Paragraph 3 (Editor A's offline edit) and Paragraph 4 (Editor B's online edit); no data is lost

**Category:** edge-case
**Priority:** must-have

### AC-5: Offline Edit on Same Range as Online Edit Resolves to Online Edit
**Given** Editor A goes offline and edits a text range; Editor B (online) edits the same range while Editor A is offline; Editor A reconnects
**When** Yjs merges the offline and online updates
**Then** Editor B's (online/server-state) version of the range is preserved; Editor A's offline edit on that range does not overwrite Editor B's version; Editor A's undo history records their offline edit so they can see what was overwritten

**Category:** edge-case
**Priority:** must-have

### AC-6: No Characters Duplicated or Dropped During Merge
**Given** any merge scenario (concurrent real-time or offline reconnect)
**When** the Yjs CRDT resolves the updates
**Then** the resulting document contains no character duplication, no dropped characters outside the conflict range, and no corrupted formatting marks at merge sites; validated by automated diff of pre-merge and post-merge document states

**Category:** edge-case
**Priority:** must-have

### AC-7: Same-Document Multiple Conflict Sites Resolved Independently
**Given** Editor A and Editor B have simultaneous conflicts at two different positions in the same document (e.g., both editing positions 10–15 and both editing positions 50–55 at the same time)
**When** Yjs resolves both conflicts
**Then** each conflict site is resolved independently by server relay order; resolution at one site does not affect the other site

**Category:** edge-case
**Priority:** must-have

### AC-AUTH-1: Unauthenticated Access Rejected
**Given** no valid authentication token is present
**When** a client attempts to submit a Yjs update to the document's sync room
**Then** the system closes the connection with code 4001 (Unauthorized); the update is not relayed to other clients

**Category:** security
**Priority:** must-have

### AC-AUTH-2: Insufficient Permissions Rejected
**Given** an authenticated user with only `view` or `comment` permission
**When** a client attempts to submit a document-content Yjs update (not just awareness/cursor data)
**Then** the y-websocket server rejects the update with close code 4003 (Forbidden); the update is not applied or relayed

**Category:** security
**Priority:** must-have

## Non-Functional Requirements

### Error Handling
| Scenario | Expected Behavior | Priority |
|----------|------------------|----------|
| Yjs update payload is malformed (invalid binary) | Server discards the update, logs a warning with client ID and document ID, does not relay to other clients, does not crash the service | must-have |
| Merge produces a document state that fails schema validation | Log the anomaly with full document ID and both update payloads; do not persist the invalid state; surface "Sync error — please reload" to both editors | must-have |

### Performance
- **Merge latency:** Yjs CRDT merge must complete within 50ms of receiving both concurrent updates (the merge overhead is part of the 500ms propagation SLA)
- **Stress test:** Acceptance tests must simulate 10 concurrent editors each editing different ranges simultaneously to verify no merge degradation

### Security
- **Server relay ordering:** The server must not allow clients to manipulate relay ordering via timestamp spoofing; ordering is determined by server receipt time only
- **No conflict escalation to UI:** Users must never see a raw Yjs conflict marker or merge ID in the editor canvas

### Accessibility
- Conflict resolution is transparent to all users; no accessibility considerations specific to merge behavior (no UI rendered)

## Open Questions
1. (OQ-1) The assumption that offline edits lose to server state (G-6) is not the behavior most users expect — many expect their offline work to win. This assumption MUST be validated with stakeholders before implementation. If the expected behavior is reversed (offline wins), AC-5 must be rewritten.
2. Should the user be notified when their edit was overwritten by a same-range conflict? ("Your edit at position X was overwritten by another editor.") The current scope has no notification, but this may surprise users.
